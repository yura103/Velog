<h3 id="1-문제-정의problem-framing">1. 문제 정의(Problem Framing)</h3>
<p>&quot;무엇을 예측할지&quot;보다 중요한 건 &quot;어떻게 쓸지&quot;다!</p>
<h4 id="1-예측-대상-정의">1) 예측 대상 정의</h4>
<ul>
<li>분류: y가 0/1, A/B/C 같은 범주</li>
<li>회귀: y가 연속값(가격, 수요, 시간 등)<h4 id="2-예측이-쓰이는-방식">2) 예측이 쓰이는 방식</h4>
</li>
<li>예측 확률을 그대로 쓰나?</li>
<li>특정 임계값으로 의사결정하나?</li>
<li>회귀 결과에 제약이 있나?<h4 id="3-평가-지표kpi-정의">3) 평가 지표(KPI) 정의</h4>
</li>
<li>분류: F1, ROC-AUC, PR-AUC, Recall@k 등</li>
<li>회귀: MAE, RMSE, MAPE, R^2 등<br />

</li>
</ul>
<h3 id="2-데이터-점검--eda">2. 데이터 점검 &amp; EDA</h3>
<p>단순히 그래프를 그리는 과정이 아닌 데이터를 이해하고 리스크를 찾는 과정</p>
<h4 id="1-타깃y-확인">1) 타깃(y) 확인</h4>
<ul>
<li>분류: 클래스 비율(불균형 여부), 데이터 드리프트</li>
<li>회귀: 분포 형태(롱테일 여부), 이상치, 음수 가능 여부<ul>
<li>y 분포가 한쪽으로 심하게 치우치면 log1p 고려</li>
<li>극단값이 손실(MSE)을 지배할 때, 이상치 처리 전략 필요<h4 id="2-피처x-점검">2) 피처(X) 점검</h4>
</li>
</ul>
</li>
<li>결측치 비율과 패턴</li>
<li>범주형/수치형 구분</li>
<li>누수 후보 피처 존재 여부
예) &quot;대출 승인 여부 예측&quot;인데 '승인 후 생성된' 피처가 들어가면 누수!<h4 id="3-데이터-단위-확인">3) 데이터 단위 확인</h4>
</li>
<li>한 row가 &quot;고객 1명&quot;인지 &quot;거래 1건&quot;인지</li>
<li>여러 row가 같은 엔터티를 공유하는지(고객 중복) -&gt; 분할 전략에 큰 영향<br />

</li>
</ul>
<h3 id="3-데이터-분할-전략">3. 데이터 분할 전략</h3>
<h4 id="1-기본-분할일반-tabular">1) 기본 분할(일반 tabular)</h4>
<ul>
<li>train/valid/test</li>
<li>데이터가 작으면 K-Fold CV 추가<h4 id="2-분류-stratified-split">2) 분류: Stratified split</h4>
</li>
<li>클래스 비율 유지가 중요하면 Stratified 사용<h4 id="3-그룹-단위-누수-방지group-split">3) 그룹 단위 누수 방지(Group split)</h4>
</li>
</ul>
<p>같은 고객의 데이터가 train과 test에 동시에 있으면 누수!
예) 고객별 여러 거래 row가 있는 데이터 -&gt; 고객 단위로 GroupKFold/GroupShuffleSplit</p>
<h4 id="4-시계열-시간-기준-분할이-정석">4) 시계열: 시간 기준 분할이 정석</h4>
<ul>
<li>랜덤 셔플 금지</li>
<li>과거로 학습 -&gt; 미래를 평가<br />

</li>
</ul>
<h3 id="4-전처리--피처-엔지니어링">4. 전처리 &amp; 피처 엔지니어링</h3>
<p>전처리는 &quot;성능 올리는 기술&quot;이기도 하지만 그보다 중요한 건 train과 inference에서 동일하게 재현되는 것!</p>
<h4 id="1-전처리">1) 전처리</h4>
<ul>
<li>결측치 처리: mean/median, &quot;missing&quot;카테고리, 모델 내 처리</li>
<li>범주형 인코딩: one-hot/target encoding/catboost 내장</li>
<li>스케일링: Standard Scaler, MinMaxScaler -&gt; 선형/SVM/KNN에서 특히 중요</li>
<li>이상치 처리: clip/winsorize/robust scaling<h4 id="2-피처-엔지니어링">2) 피처 엔지니어링</h4>
</li>
<li>수치형 변환: log, binning</li>
<li>상호작용: ratio, diff, product</li>
<li>집계 피처(거래 데이터): 고객별 count/sum/mean, 최근 N일 rolling 등</li>
<li>시간 피처: 요일/월/시간대, lag, rolling mean<br />

</li>
</ul>
<h3 id="5-baseline-모델">5. Baseline 모델</h3>
<p>성능 좋은 모델을 사용하기 전에 누수/피처 유효성/개선 여지를 확인하는 기준</p>
<ul>
<li>모델<ul>
<li>분류: Logistic Regression, Linear SVM</li>
<li>회귀: Linear/Ridge, RandomForestRegressor</li>
</ul>
</li>
<li>확인할 것<ul>
<li>점수가 말이 되는가? -&gt; 누수 의심</li>
<li>어떤 피처가 중요한가? -&gt; 해석 가능한 모델에서 확인</li>
<li>단순한 규칙으로도 해결 가능한 문제인가?<br />

</li>
</ul>
</li>
</ul>
<h3 id="6-강력-모델--튜닝">6. 강력 모델 + 튜닝</h3>
<h4 id="1-기본-루트">1) 기본 루트</h4>
<ul>
<li>tabular: XGBoost/LightGBM/CatBoost가 1순위 되는 경우가 많음.</li>
<li>데이터가 작거나 해석이 중요하면 선형 모델 + 좋은 피처로 끝내기도 함.<h4 id="2-튜닝-순서">2) 튜닝 순서</h4>
</li>
</ul>
<ol>
<li>learning_rate 낮게 고정(0.05 ~ 0.1)</li>
<li>depth/num_leaves로 모델 복잡도 설정</li>
<li>min_data_in_leaf로 과적합 방지</li>
<li>subsample/feature_fraction으로 랜덤성 추가</li>
<li>n_estimators 늘리되 early stopping 사용</li>
</ol>

<h3 id="7-검증--오류-분석">7. 검증 &amp; 오류 분석</h3>
<h4 id="1-cv-점수-vs-holdout-점수-비교">1) CV 점수 vs Holdout 점수 비교</h4>
<ul>
<li>CV는 높은데 holdout이 낮으면 분할 전략/누수/드리프트 의심</li>
<li>둘 다 낮으면 피처/모델 표현력 부족<h4 id="2-오류-분석">2) 오류 분석</h4>
</li>
<li>어떤 구간에서 오차가 커지는지</li>
<li>어떤 피처 값에서 틀리는지</li>
<li>시간에 따라 성능이 떨어지는지 (드리프트)</li>
<li>회귀면: &quot;큰 값에서만 크게 틀리는지&quot; -&gt; log 변환 힌트<h5 id="오류-분석과-하이퍼파라미터로-모델-개선-가능">오류 분석과 하이퍼파라미터로 모델 개선 가능</h5>
<br />

</li>
</ul>
<h3 id="8-최종-모델-선택-및-해석">8. 최종 모델 선택 및 해석</h3>
<h4 id="1-단일-모델-vs-앙상블">1) 단일 모델 vs 앙상블</h4>
<ul>
<li>단일 부스팅 모델이 대부분 충분히 강함</li>
<li>리더보드/경쟁형이면 stacking으로 더 올릴 수 있음</li>
<li>실무 배포는 단순한 모델이 유리할 때도 있음<h4 id="2-해석">2) 해석</h4>
</li>
<li>선형: 계수</li>
<li>트리/부스팅: feature importance, SHAP</li>
<li>&quot;왜 틀리는지&quot;를 아는 게 중요함!<br />

</li>
</ul>

<li>모델/전처리 버전 관리<br />

</li>
</ul>
